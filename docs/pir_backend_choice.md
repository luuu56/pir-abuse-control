# 盲签方案选型定稿 (Blind Signature Choice)

## 1. 主线方案：RSA 盲签名 (RSA Blind Signature)
在原型开发的第一阶段，我们选择 **RSA 盲签名** 作为核心票据签发机制。
* **技术实现**：第一版原型基于 Python 的 `pycryptodome` 提供 RSA 基础能力，并在其上实现完整的盲化（Blinding）、签发（Signing）、去盲（Unblinding）与验证（Verification）流程。
* **选择理由**：
  1. **工程成熟度**：RSA 盲签名机制经典且直观，便于在开发初期快速打通全链路逻辑。
  2. **验证成本可控**：Verifier 侧的公钥验签成本相对较低，适合作为用户态前置验证链的核心环节。
  3. **安全性验证**：在正确的盲签协议实现前提下，2048-bit RSA 足以支撑第一版原型对不可伪造性与“签发—使用”不可直接链接性的工程验证。

## 2. 备选方案与演进
* **方案演进**：第一阶段稳定后，可评估是否引入基于 secp256k1 的盲 Schnorr 签名，用于对比不同数学基础下的票据尺寸与计算开销。

## 3. 核心依赖与风险控制
* **主要依赖**：`pycryptodome`。
* **风险规避**：若在开发过程中发现 `pycryptodome` 的高层 API 对盲签协议细节支持受限，将考虑引入更底层的密码学支持或自定义协议封装。
* **实现边界**：明确盲签的完整验签逻辑位于用户态 Verifier，eBPF 层不涉及非对称加密运算。
